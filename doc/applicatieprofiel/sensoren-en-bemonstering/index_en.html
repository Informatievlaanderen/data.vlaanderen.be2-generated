This document describes an application profile, in this case Sensors and Sampling (Application Profile). The application profile specifies a semantic data model covering a set of usecases. The specification consists of terms with their constraints (cardinalities, codelists, usage guidelines).

Summary

The Sensors and Sampling application profile explains how the terminology should be used to exchange data on observation instruments with which Observations are carried out and on the activity of obtaining representative spatial or physical samples.

This application profile is based on the W3C SSN ontology and its extensions. It also builds on the OSLO Observations and Measurements application profile.

Observation is taken from Observations & Measurements with its characteristics, but Process (this is the Procedure used in an Observation) is split into Sensor (the instrument with which the observation is made) and Observation Procedure (the method by which the Observation is made).

Sampling objects (location and Sample) are also still there but are now the result of a Sampling (activity), just like Observation (which is also an activity) with a Sampler (the instrument with which the Sampling is done) and a sampling Procedure (the method followed when Sampling).

Observations can now also be grouped in an Observation Collection, possibly specifying the Observation attribute on which they are grouped (e.g. the same location or the same observed Characteristic).

Sensors are a specialization of System, which allows to specify the characteristics ve Sensor, e.g. the resolution or the measuring range.

A Device is also added as a subclass of System, which allows a Sensor (which is also a System) to be viewed as a device (with a brand name or a manufacturer, for example).

Property Expects Type Cardinality Description Usage Codelist

Property Expects Type Cardinality Description Usage Codelist
